# アーキテクチャ方針

## 1. 推奨アーキテクチャ
- フロントエンド: Next.js（App Router想定）
- BFF/API: Next.js Route Handler / Server Actions（要確認）
- DB/Auth: Supabase
- 配備: Vercel

## 2. Next.js構成
- UI層: 表示・入力
- Application/Service層: ユースケース（献立提案、価格集計等）
- Domain層: 計算ロジック
- Infrastructure層: Supabaseアクセス

## 3. Supabase構成
- Auth: MVP 0から必須
- PostgreSQL: user_id付きテーブルを標準
- RLS: 全ユーザーデータテーブルで有効化

## 4. 認証方針
- MVP 0: Googleログイン優先
- 将来対応: メール/Appleログイン
- 理由: 個人の購買・食事履歴を保護するため

## 5. RLS方針
- `auth.uid() = user_id` を基本
- 共通マスタは読み取り公開、更新は制限（要確認）
- データ削除は本人のみ

## 6. AI利用方針
- AIが担当: 提案・説明（レシピ案、予算改善案など）
- AIが担当しない: 予算計算、予想価格、ルール判定、数量集計
- AI障害時は非AI機能が継続する設計

## 7. サービス層の責務
- 献立提案サービス
- 買い物リスト生成サービス
- 価格計算サービス
- 栄養計算サービス
> UI層への直書きを禁止。

## 8. セキュリティ方針
- 入力バリデーション（Zod想定）
- 機微情報の最小ログ化
- AI API送信データの最小化と同意設計（将来対応）

## 9. エラーハンドリング方針
- ユーザー向け: 行動可能なメッセージ
- 開発者向け: 原因分類（認証/権限/バリデーション/外部依存）
- 未確定: エラーコード体系

## 10. ログ方針
- 操作イベントとエラーを分離
- 食事/価格/健康データ本文はログ出力しない

## 11. 将来の拡張方針
- 家族共有（組織・ロール）
- 管理画面の段階導入
- 外部連携（Appleヘルスケア等）
